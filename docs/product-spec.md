# Codehive - Product Specification

Persistent AI coding agent workspace with sub-agent orchestration, accessible from web, mobile, Telegram, and terminal.

## Vision

A platform for managing AI coding agents that live in projects, execute tasks autonomously, create sub-agents, and are accessible from multiple interfaces. Not a chat — an operating system for agent sessions.

**One-liner:** "Persistent coding agent workspace: agents live in projects, work task queues, accessible from web/mobile/Telegram/terminal, show transparent diffs, execute locally or via SSH."

## Core Entities

### Workspace
Root-level container on the server.
- Contains all projects
- Global settings, integrations, secrets, agent library
- SSH access and tunnel management

### Project
A codebase (git repo or folder) with its own development context.
- Issue tracker (project-level tasks)
- Knowledge base (tech stack, architecture, conventions, decisions)
- List of sessions
- Project-specific agent configurations
- Open Decisions — unresolved questions tracked explicitly

### Issue
Long-lived project task (like a GitHub Issue).
- Can have multiple sessions working on it
- Syncs bidirectionally with GitHub Issues
- When a session completes, it can create a PR and close the issue

### Session
A live AI agent working on a task. The central unit of the system.
- Chat history
- ToDo list (task queue)
- Timeline of actions
- Changed files / diffs
- Sub-agent tree
- Checkpoints
- Current mode (see Agent Modes)
- Status: `idle` | `planning` | `executing` | `waiting_input` | `waiting_approval` | `blocked` | `completed` | `failed`
- Linked to an Issue (optional)
- Can be a sub-agent of another session (`parent_session_id`)

### Session ToDo
Operational task queue for the agent within a session.
- Prevents the agent from idling when there's work to do
- Agent auto-picks the next task when current one completes
- Can be created by the user or generated by the agent from a plan

Settings per session:
- Queue enabled/disabled
- Auto-start next task
- Require approval before destructive actions
- Stop on failure / continue on failure
- Concurrency

Per task:
- Priority, dependencies, manual approval flag, retry policy, timeout, done criteria
- Status: `pending` | `running` | `blocked` | `done` | `failed` | `skipped`
- Created by: user or agent

### Pending Questions
When the agent encounters a non-blocking question and the user is unavailable:
- Question is saved to a queue
- Agent continues with the next task
- User answers later from any client

Question handling priority:
1. Agent can decide autonomously (using project knowledge) — proceeds
2. Question is non-blocking — deferred to Pending Questions
3. Question is critical/blocking — session pauses (last resort)

## Agent Modes

Sessions support multiple cognitive modes that change agent behavior.

### Brainstorm
Free-form ideation. Agent asks open questions, proposes alternatives, identifies gaps. Doesn't rush to implementation.

### Interview
Structured requirements gathering. Agent asks batched questions (3-7 per batch), saves answers to project knowledge. Produces a project spec.

### Planning
Turns ideas into structure: milestones, sessions, tasks, missions. Shows what's decided vs. what's open. Doesn't write code.

### Execution
Standard coding mode. Agent edits files, runs commands, executes tasks, creates sub-agents.

### Review
Evaluates what's been done. Checks quality, proposes improvements, prepares next steps.

Modes switch fluidly within a session: brainstorm -> interview -> planning -> execution -> review.

## Sub-Agents

Sub-agents are full sessions with a parent, not opaque tool calls.

### Structure
```
Session A (orchestrator)
   +-- Session B (sub-agent: backend)
   +-- Session C (sub-agent: frontend)
   +-- Session D (sub-agent: tests)
```

Each sub-agent has its own chat, ToDo, diff, timeline, and logs — all visible to the user.

### Key Principle
**A sub-agent mission must be completable in one focused session.**

Sub-agents receive a *mission*, not a backlog. If a mission is too complex, the system suggests decomposition via a planner agent.

### Communication
Sub-agents return structured reports to the orchestrator:
- Status (completed/failed/blocked)
- Summary of changes
- Files changed
- Test results

The orchestrator aggregates progress and decides next steps (spawn more agents, fix issues, merge results).

### Visibility
- Tree view of all sub-agents with status
- Click into any sub-agent to see full session
- Aggregated progress in the orchestrator view
- Unified project timeline across all sessions

## Agent Roles

### Global Role Library
System-wide roles that define agent behavior:
- **Developer** — implements features, modifies code, writes tests
- **Tester** — writes and runs tests, verifies functionality
- **Product Manager** — decomposes tasks, manages priorities
- **Research Agent** — investigates options, produces reports
- **Bug Fixer** — reproduces, diagnoses, and fixes bugs
- **Refactor Engineer** — improves code structure

Each role defines: responsibilities, allowed tools, coding style, task types.

### Project-Level Customization
Projects can override global roles with project-specific versions:
```
Developer (global)
   -> Python Developer (project override)
```

No extension/inheritance chains — just global roles and simple per-project overrides.

### Custom Roles
Users can create custom roles at both levels — global (available to all projects) or project-specific (e.g., Security Auditor, Performance Optimizer).

## Project Archetypes

Pre-defined templates for common project types.

### Examples
| Archetype | Agents | Workflow |
|-----------|--------|----------|
| Software Development | PM, Developer, Tester | Idea -> Planning -> Implementation -> Testing -> Review |
| Research | Researcher, Analyst, Writer | Question -> Research -> Analysis -> Report |
| Technical Operations | DevOps, Monitoring, Incident Responder | Detect -> Diagnose -> Fix -> Verify |
| Code Maintenance | Bug Fixer, Refactor Agent, Test Agent | Issue -> Investigation -> Fix -> Test |

Archetypes define: agents, roles, workflow, coding rules, tech stack.

Auto-detection: when creating a project, the system suggests an archetype based on description.

Archetypes are clonable and customizable.

## Agent Behavior (Markdown Files)

Each agent role's behavior is defined in a markdown file (similar to CLAUDE.md). These files describe the agent's responsibilities, workflow, rules, and style. No separate "playbook" abstraction needed — the agent's md file *is* its playbook.

## Autonomy & Decision Framework

### User Presence Modes
- **Interactive** — agent asks questions immediately
- **Async** — agent queues questions and continues working

### Decision Logic
When the agent faces uncertainty:
1. **High confidence + project knowledge exists** — decide autonomously
2. **Non-blocking question** — defer to Pending Questions, continue work
3. **Blocking question** — pause session (last resort)

### Agent Charter
Per-project document defining goals, constraints, tech stack, coding rules, and decision policies. The agent consults this when making autonomous decisions.

## Transparency & Observability

### Live Diff
Real-time view of files being changed: line-level diffs, pending vs. applied.

### Change Timeline
Chronological log of every action:
```
10:04  planning tasks
10:05  analyzing repo
10:06  editing auth_service.py
10:07  running tests
10:08  fixing failing test
```

### Semantic Summary
Human-readable descriptions: "added auth middleware", "updated imports".

### Approval Gates
Confirmation required before: file deletion, force push, migration apply, production commands, secret-related edits.

### Session Replay
Open any completed session and step through the full history of actions.

### Persistent Logs
All sessions store: agent messages, tool calls, file changes, terminal output, diff history. Enables debugging, performance analysis, and future agent training.

## Checkpoints & Rollback

### Level 1 — Git Commits
After each significant operation, the agent commits. Standard git rollback.

### Level 2 — Session Checkpoints
Snapshots including: git state, open tasks, session memory, environment. Can restore to any checkpoint.

### Level 3 — System Rollback
For when the agent breaks the system itself. Maintains a "last known good version".

### Self-Modification Safety
When the agent modifies codehive's own code:
- Mandatory checkpoints
- Separate branch
- No merge without user approval

## Interfaces

### Web / Desktop (Rich Client)
Primary interface. Full access to all features: project management, session interaction, diff viewer, sub-agent graphs, planning boards.

### Mobile App (Control Tower)
Not an IDE. Optimized for:
- Monitoring session progress
- Receiving notifications ("approval needed")
- Answering pending questions
- Voice input
- Viewing diff summaries
- Quick task creation

### Telegram Bot (Lightweight Client)
Minimal control surface:
- Check status
- Answer questions
- Start sessions
- Approve/reject actions
- View summaries

Not for: reading large diffs, deep debugging, interactive coding.

### CLI
Non-interactive command-line interface for scripting and quick operations:
- `codehive projects list`, `codehive session create`, `codehive session send`, etc.
- Usable in scripts, cron jobs, CI/CD pipelines
- All core operations available without a UI

### Terminal Client (TUI — Emergency + Daily Use)
Interactive TUI accessible via SSH directly on the server. Three roles:

**1. Full working client** — projects, sessions, chat, ToDo, sub-agents, timeline, approvals

**2. Operator console** — active sessions, stuck agents, errors, tunnel status

**3. Rescue mode** — stop runaway sessions, kill agents, rollback checkpoints, restart services, maintenance mode

Designed to work on a phone screen over SSH in emergencies.

**Design principle:** Terminal-first survivability. The system must be fully manageable from the terminal even when all other interfaces are degraded.

### Shared State
All clients operate on the same state model. The same session is accessible from web, mobile, terminal, and Telegram simultaneously.

## Integrations

### GitHub
- Import repositories as projects
- One-way issue sync: GitHub Issues -> codehive internal tracker
- Issue creation in GitHub triggers session creation in codehive
- Bidirectional sync and PR creation from sessions — future addition

### Claude Code
Bridge integration using the official Claude Code CLI (not API key proxying).
```
codehive -> local claude-code CLI -> Claude
```
Session engine modes:
- `engine: claude_code` — uses Claude Code subscription
- `engine: native` — uses codehive's own agent

Same UI, task management, and diff viewer for both engines.

### Voice Input
Input adapter (not core logic):
- Voice capture -> STT -> text -> route to appropriate chat level
- Transcript preview before send
- Destructive actions from voice require text confirmation

## Security

### Filesystem Sandbox
- Agent confined to project root
- No symlink escapes
- Policy on hidden/system directories

### Command Policy
- Allowlist / denylist / confirmation per command type
- Network access: configurable on/off
- Package install: ask/auto
- Git push: always ask
- Destructive shell: always ask

### Secrets
- No raw keys in logs
- Environment variables masked
- stdout/stderr redacted
- Access to secrets audited

## Architecture (High Level)

### Control Plane
API, auth, projects, sessions, tasks, events, notifications, UI adapters.

### Execution Plane
Separate workers: shell, file ops, git, SSH tunnels, test runner, diff service.

UI can crash — sessions keep running. Telegram doesn't depend on web lifecycle.

### Session Runtime
```
Session Runtime =
  context + filesystem + shell + git
  + task queue + memory/state
  + event stream + approval policy
  + model adapter
```

Any surface (web, mobile, Telegram, terminal) connects to the same runtime.

### Event Stream
Both engines emit uniform events:
`message.created`, `tool.call.started`, `tool.call.finished`, `file.changed`, `diff.updated`, `task.started`, `task.completed`, `approval.required`, `session.waiting`, `session.failed`

UI is engine-agnostic.

## New Project Flow

1. User says "I want to start a new project about X"
2. System offers: Brainstorm / Guided Interview / Spec from Notes / Start from Repo
3. Brainstorm or Interview collects requirements
4. Agent produces: Project Brief, Open Decisions, Suggested Milestones, First Sessions
5. User approves
6. Working sessions launch with appropriate archetypes and playbooks

## Key Product Principles

1. **Sessions, not chats** — agents are persistent workers with state, not disposable conversations
2. **Issue = why, Session = who, ToDo = what** — clean mental model
3. **Terminal-first survivability** — system manageable from SSH even when everything else is down
4. **Transparent agents** — every action visible, replayable, rollbackable
5. **Mission-sized sub-agents** — small focused tasks, not vague backlogs
6. **Policy replaces presence** — when user is away, decision framework keeps agents productive
7. **Platform over brain** — agent intelligence is swappable, the orchestration platform is the product
